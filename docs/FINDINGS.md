# 弱LLM向け 委譲タスク指示テンプレート

対象: ローカル中級モデル(Qwen3.6-27B 等)に claude-local 経由でコーディングを委譲するとき。
根拠: 2026-07-08 の実測(weak-llm-lab)。曖昧指示は詳細指示に対し 入力1.5x・出力2.3x・往復1.3x のコスト、かつ品質は自己判断で発散。

## 原則(なぜ効くか)
弱LLMは「明示スペックの実装」は上位モデル並みにこなすが、「曖昧さの補完」でウロウロして発散する。
→ 考える部分(何を・どう検証するか)は発注側(上位モデル/人間)が固め、弱LLMには実装だけをやらせる。

## テンプレート(5ブロック)

### 1. 目的と成果物(1文)
「何を作るか」を1文で。ファイル名まで指定する。
例: 「calc.py に evaluate(expr:str)->float を実装する」

### 2. 仕様(箇条書き・番号付き)
入力形式・出力項目・制約を漏れなく列挙。弱LLMは書いていないことは自己判断で埋めるので、
「埋めてほしくない箇所」は全て書く。
- 入力: <形式・ソース>
- 出力: <項目を全列挙>
- 制約: <禁止事項(例: eval禁止、外部依存禁止)、エラー処理の方針>

### 3. 決定権の範囲(明示)
「自由に決めてよい / 必ずこの通り」を分ける。これがないと勝手に仕様を確定させる。
例: 「変数名・内部構造は自由。ただし関数シグネチャとエラー型は指定通り」

### 4. 自己検証手順(成功条件を埋め込む)
成功を機械的に確認できる手順を指示に入れる。弱LLMに自分でテストを書かせ・実行させる。
例: 「test_X.py に正常系N・異常系Mのpytestを書き、python3 -m pytest で全通過を確認」

### 5. 出力契約(余計な生成を抑制)
「完了したら DONE とだけ報告」等。長い自己解説を抑え出力トークンを節約。

## アンチパターン(実測で確認した失敗モード)
- 探索迷子: 曖昧指示だと ls から延々さまよう → 目的とファイル名を最初に固定する
- 暗黙の自己判断: 決定権が不明だと検証不能な成果物になる → ブロック3で範囲を切る
- 過剰実装: 指定がないと余計な機能・行数を盛る → 仕様を「これだけ」と閉じる

## 発注側(上位モデル)の責務
- タスクを渡す前にスペックを完成させる(スペック作成は数千tokで安い)
- 戻ってきた成果物は独立に検証する(弱LLMの自己申告を信じない)
- 探索が必要な曖昧タスクは、まず自分でスペック化してから委譲する

## 実測ログ(2026-07-08, Qwen3.6-27B-NVFP4 / claude-local)

| 実験 | タスク | 指示 | 入力tok | 出力tok | 品質 |
|---|---|---|---|---|---|
| 1 | 数式パーサ | 詳細+検証 | 96,558 | 1,126 | 完璧(pytest8+隠し5全通過) |
| 2A | ログ集計 | 曖昧1行 | 187,538 | 5,124 | 動くが仕様全て自己判断・323行 |
| 2B | ログ集計 | 詳細+契約 | 126,019 | 2,262 | 仕様6項目充足 |

| 4D | LRU | テンプレ準拠 | 96,102 | 954 | 隠し7/7 + 自作テスト7件 |

### 結論
- 曖昧さのコストは非有名タスクで顕在化(実験2: 曖昧は詳細の入力1.5x/出力2.3x、品質は発散)。
- 有名・定型タスク(LRU等)はモデルの素振り済みで、素の指示でも品質が出る(実験4: C/D差は小)。
  → テンプレの価値は「モデルが迷う非定型タスク」で最大。定型タスクには過剰。
- テンプレの副次効果: 自己検証手順を入れると弱LLMが自分でテストを書き実行する(4Dで7件自作)。
  発注側の検証コストを下げる。
### 運用指針
タスクを「モデルにとって定型か/非定型か」で振り分ける:
- 定型(有名アルゴリズム、CRUD、定番変換) → 素の指示でよい。テンプレは出力契約だけ足す。
- 非定型(独自仕様、プロジェクト固有ロジック、曖昧な要件) → フルテンプレ必須。

## 深掘り: subtle correctness の境界(2026-07-08 追加, 実験5-7)

| 実験 | タスク | 罠の種類 | 罠の明示 | 入力tok | 隠しテスト |
|---|---|---|---|---|---|
| 5E | semver比較 | standard・prerelease優先順位 | なし(semver.org準拠のみ) | 214k | 罠7/7 全通過 |
| 6G | 区間マージ(gap許容) | bespoke・自然な既定解釈 | なし | 72k | 罠7/7 全通過 |
| 7H1 | 区間マージ(接触=非統合) | bespoke・反直感(標準の逆) | 一度・一例で明示 | 119k | 罠5/5 全通過 |

### 深掘りの結論
- 当初仮説「bespokeのsubtleは罠を明示しないと失敗」は棄却。真の軸は standard/bespoke でも
  subtleさでもなく「**明示されているか**」。
- 弱LLMは書いた規則には従う。反直感ルール(接触は統合しない=標準マージの逆)ですら、
  一度・一例で書けば訓練データの標準挙動に引きずられず正答した。
- 失敗するのは「書いていないこと」= 曖昧(実験2)と 非既定ルールの省略。後者は、発注側が
  逸脱点を言い落とすとモデルが既定(標準挙動)で「正しく」実装し意図とズレる形で顕在化する。
- よって発注側の責務は「難しさの見極め」より「逸脱点の言い落としゼロ化」。standard×既定通りは
  薄い指示でよく、過剰スペックは無駄。

## 既定プローブ・ハーネス(2026-07-08, 粒度問題への回答)
道具: `default_probe.py`(スキル weak-llm-playbook/scripts に同梱)。
20判断点を各N回投げ、生成コードを実行して既定と安定性を測る。JSON保存+ `--diff` 対応。

### Qwen3.6-27B 既定プロファイル(20点, N=5)
揺れる=必ず明示: avg_empty(例外/0), fib_index(定義ゆらぎ)。
安定だが意図とズレやすい: nth_index=0-indexed / sort_case=ASCII順 / range_incl=終端含む /
missing_key=KeyError / div_zero=例外 / none_input=例外。
安定で直感的: interval_touch=接触も統合 / round_half=2.5→3 / dedup=順序保持 / split_multi=空白畳む /
empty_split=空リスト / flatten=深く平坦化 / reverse=昇順 / title_case=各語頭 / neg_modulo=python式。

### 粒度問題の最終回答
明示すべき判断点 = 「安定性が低い ∪ 既定が意図と食い違う」だけ。それ以外は書かない(過剰は無駄)。
①はプローブが自動検出、②は既定の中身を提示して発注側が照合する。

### モデル別検知
`--diff A.json B.json` でモデル間の既定差を抽出=「モデルを替えたら書き換える点」。
合成モデルで検証済み(反転3点を★、揺れ1点を△で正しく抽出)。
※実モデル比較(step3p7/minimax/gemma)は保留: 各バックエンドの conda env が消失
  (vllm / vllm-step)、または両GPU必須でQwenと競合、GLMは手起動要。env修復は別タスク。
  思考モードは1リクエスト2000tok級で20点×5回が非現実的+同一重みのため真の別モデルでない→不採用。

## 実モデル間 diff 成立(2026-07-08, タスク完遂)
判断点を32点に拡張し、Phi-3.5-mini(3.8B)を m3vllm環境でGPU1に配信(Qwen無傷)して実測。
起動の罠: FlashInferサンプリングJITがnvcc未検出で落ちる→ `VLLM_USE_FLASHINFER_SAMPLER=0` +
`--enforce-eager` で回避(スクリプト /var/hf/dl_probe_models/serve_phi.sh)。

### Qwen3.6-27B vs Phi-3.5-mini の既定差(9/32点)
★既定が違う: dedup_order / range_incl / split_multi / missing_key / date_fmt / max_empty / dup_count
  - missing_key: Qwen=KeyError vs Phi=None返す ← 同じ「辞書から値を取る」でも真逆
  - range_incl: Qwen=終端含む vs Phi=終端含まず
  - date_fmt: Qwen=ISO(0埋め) vs Phi=スラッシュ区切り
△不安定(弱いPhiで顕著): dedup/split_multiはPhiがNameError/ModuleNotFoundErrorを出す(安定0.4)
  = 弱いモデルほど「そもそも正しく実装できない=明示しても危うい」点が増える。

### 結論(モデル別検知の実証)
- 同じ意図でも、明示すべき点はモデルで入れ替わる(9/32が差分)。prof をモデルごとに1回取り、
  `--diff` すれば「乗り換え時に書き換える点」が機械的に出る。
- 弱いモデルは「既定が違う」だけでなく「安定して実装できない」点が増える。後者は明示だけでは
  救えず、タスク自体を避けるか粒度を上げる(plan-and-delegateのLv昇降)判断材料になる。

## 改善2件(2026-07-08)

### (a) プローブの予測妥当性を検証: 直叩き ≒ エージェント委譲
懸念: プローブはvLLM直叩きで既定を測るが、実委譲はclaude-local(Claude Codeのsysプロンプト+
エージェントループ)経由。両者がズレたらプローブは的外れを測っている。
検証: Qwenの直叩き既定が明確な3点(missing_key=KeyError/range_incl=終端含む/nth_index=0-indexed)
を、仕様を伏せてclaude-localに実装させ挙動を照合 → **3/3 的中**。
結論: 安いvLLM直叩きプローブは高価なエージェント委譲の挙動を予測できる=代理指標として妥当。

### (b) 適応的リサンプリング(解像度向上)
N=5では安定性が0.6/0.8/1.0の粗い値しか出ず境界判定が甘い。安定性が曖昧帯(0.5〜0.85)の点だけ
追加サンプルを取り最大N=15まで精密化。無駄打ちなし(安定点はN=5のまま)。
効果(Qwen): avg_empty 0.6→0.53(N15, 例外8/0返す7=真の五分五分と確定) /
max_empty 0.8→0.86(N7, 実は安定側と判明) / dup_count 0.53(真に不安定+分類器の弱さも露呈)。
= 「必ず明示すべき真に揺れる点」と「実は安定な点」を切り分けられるようになった。

## スキルの操作者モデル依存性の検証(2026-07-09)
問い: weak-llm-playbook を実行する側(オーケストレータ)がFable 5でなくOpus/Haikuでも機能するか。
方法: 同一スキル+Qwen既定プロファイルを Opus と Haiku のサブエージェントに読ませ、意図がQwen既定と
逆になる2点(欠損キー→None[既定KeyError]、範囲→半開[既定終端含む])+一致1点(不足時全件[既定一致])
を仕込んだ委譲タスクのスペックを書かせて採点。

結果:
- Opus: 満点+α。2ズレをピン留め、一致点はテストのみ、さらに未設計の曖昧さ(「上位n件」=ソートか
  先頭nか)を自力検知+弁別例で固定、囮点(nth_index/none_input)を関連性で正しく除外。発注側の水準。
- Haiku(下限): 設計3点の②照合は全正解(プロファイル正確引用)。傷2つ=一致点をスペック本文に冗長記載
  (分析では不要と正答)、未設計の曖昧さは見逃し。核心は外さず。

結論(設計原理): スキルが判断を「機械的手順+データ(プロファイル)」に外部化しているため、②照合
(既定vs意図の突き合わせ=「2値の比較」)は Haiku でも遂行できる=操作者モデルへの移植性が高い。
能力が効くのは「プロファイルに無い新規の曖昧さの検知」「判断点の関連性推論」「内部整合性」——
Opusは処理、Haikuは一部取りこぼす。実務含意: 弱い操作者ほどプロファイル(プローブ点)を厚くして
操作者自身の推論に委ねる部分を減らす。強い操作者はプロファイルを叩き台に新規曖昧さも拾う。

## 改善3: タスク駆動スペック穴検出器 spec_holes.py(2026-07-09)
動機: 操作者検証で残った唯一の能力依存=「固定バッテリーに無いタスク固有の曖昧さの検知」
(Opusは推論で捕捉、Haikuは見逃した)を機械化する。

方式(disagreement probing): ドラフト仕様をワーカーにK回実装させ、発注側が用意した
プローブ入力で全実装を実行。挙動が割れた入力=仕様の穴として自動報告。3シグナル:
[発散]=必ず明示 / [合意]=意図と照合のみ / 実装不能率=委譲可否。

実証(GLM-5.2をワーカー、top_nドラフト仕様):
- 「上位n件」のソート/先頭曖昧性を機械検出([3,2]×3 vs [1,2]×1) = Haikuが見逃した穴
- 想定外の第2の穴「全件返す時の順序」([2,1] vs [1,2])まで検出
- 合意3件(空/n=0/単一)は照合リストに分離
開発中の知見:
- ワーカーにエッジ入力を提案させるのは脆い(GLM-5.2は`[`反復で崩壊)→発注側供給を正に
- 全入力で例外の壊れ実装は発散分析から除外(ノイズ源)
- 副産物: GLM-5.2-VQ(:8001)はコード生成の実装不能率17-55%と不安定(Qwenよりかなり悪い)。
  spec_holes はモデル健全性チェックとしても機能する
※この日、:8000のQwenは停止され :8001 に glm-5.2 が起動されていた(ユーザー側の切替)。
