# weak-llm-fix — ドラフトを弱LLM向けに正常化して返す

引数 = ドラフトプロンプト(インラインテキスト or ファイルパス)。
成果物 = **そのまま弱LLMに投げられる完成プロンプト**(ファイル保存+全文提示)。

道具はすべて `同じskillsディレクトリの weak-llm-playbook/scripts/ ` にある(以下 `<scripts>`)。
作業ファイルは /tmp か $CLAUDE_JOB_DIR/tmp に置く。

## 手順

1. **受け取り**: 引数がファイルパスならそれを読み、インラインなら draft.txt として保存。
2. **対象外ガード(最初に判定)**: ドラフトが「関数/コードの実装」でも「文書からの抽出・整形」
   でもない場合(探索・調査・MCP/ツール利用・多段のエージェント作業)、このスキルの対象外。
   委譲せず「弱LLM委譲に向かない(理由1行)」と伝えて通常どおり自分で対応する。
3. **接続先の確定**: $PROBE_BASE → 無ければ既知ポートを疎通確認
   (`for p in 8000 8002 8003; do curl -s -m2 http://localhost:$p/v1/models; done`)。
   全滅ならユーザーにサーバー起動を依頼して止まる(勝手にモデルを起動しない)。
4. **プローブ入力の自作(このスキルの核心)**: ドラフトを読み、解釈が割れそうな点
   (境界値・空・欠損・範囲・重複・同点・型)を狙った入力を5〜8個、自分で書いて
   inputs.json に保存する。コードタスク=引数の組の配列 / 抽出タスク=入力文書(文字列)の配列。
5. **正常化の実行**:
   `python3 <scripts>/spec_holes.py draft.txt inputs.json --fix`
   (モード・関数名・モデルは自動判別される)。exit 1(穴が残った)なら、残存穴を
   fixed.txt に自分で1行ずつピン留めして再実行し、穴ゼロまで収束させる(最大2周)。
6. **意図レビュー(機械にできない部分)**: fixed.txt のピン行をドラフトの文面と突き合わせる。
   ドラフトの意図と矛盾しそうなピン(例: 「上位」なのに先頭n件でピンされている)は
   自分で書き直して再検証。判断がつかないピンは**要確認**として最終報告に明示する。
7. **モデルの癖の注入**: 対象モデルのカード/プロファイル
   (`同じskillsディレクトリの weak-llm-playbook/scripts/ profile_*.json` や repo の cards/)があれば、
   このタスクに関係する既知の罠だけ1〜2行追加する。代表例(Qwen系):
   抽出なら「生のJSONのみ・コードフェンス禁止」「欠損フィールドは null」、
   範囲表現の扱い、字数制限は守れないので文数指定に置換。無ければ省略(でっち上げない)。
8. **出力**: 完成プロンプト全文をコードブロックで提示し、保存パスを添える。
   要約は3行以内: ピン留めした点 / モデルの癖として足した行 / 要確認の点。

## 注意
- プローブは生成コードをそのまま実行する。信頼できないエンドポイントには使わない。
- ピンの中身を勝手に「良かれと思って」変えない(変えるのは意図レビューで矛盾が明確な場合のみ)。
