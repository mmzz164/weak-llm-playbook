# weak-llm-playbook — 弱LLMに何を・どう投げるかの判断と指示整形

## このスキルの位置づけ
- **判断層**: 目の前のタスクを弱LLMに単発委譲してよいか、どの詳細度で書くかを決める。
- **単発委譲**: 1ファイル〜数十行規模を1回投げて完成させる。下の5ブロックテンプレで書く。
- **大規模タスク**: 多ユニット・多ファイル・反復採点が要るものは、別途オーケストレーション基盤
  (設計書→ワーカー起動→独立採点のループ)を使う。このスキルはその前段の判断を担う。

## 起動時チェック
1. ワーカー基盤の疎通: `curl -s <base_url>/v1/models` 等で対象エンドポイントの稼働を確認。
2. 検証方法(テストコマンド)を確定。弱LLMの自己申告は信じない。採点は発注側が独立に回す。

## ステップ1: タスクを分類する(最重要)
実測で分かった弱LLMの失敗要因は「難しさ」でも「subtleさ」でもなく、**「明示されているか」**の一点。

**弱LLMは、明示された規則なら subtle でも反直感でも正しく実装する。**
- standard タスク(semver/LRU)は罠(prereleaseの優先順位)を明示せずとも記憶で正答。
- **反直感ルール**(標準は接触区間を統合するが「接触は統合しない」と逆指定)ですら、
  一度・一例で明示すれば標準挙動に引きずられず正答した。
  → 「訓練データの標準挙動に負ける」失敗は、規則を書いていれば起きない。

**本当の失敗要因は2つ、どちらも「書いていないこと」:**
- **曖昧(未記述の仕様)**: 「集計するCLI作って」の類。探索で発散し勝手に仕様を確定。
  実測: 曖昧は詳細の入力1.5x/出力2.3x、品質も低下。上位モデルより曖昧さのコストが大きい。
- **省略(未記述の非既定ルール)**: 既定と異なる挙動を書き忘れると、弱LLMは既定を選ぶ。
  モデルは標準挙動で「正しく」実装し、意図と食い違う。

### 実務ルール(分類より優先)
1. **既定(=素直な読みで誰もがそう作る挙動)から外れる規則は、全て明示する。一例を添える。**
2. **曖昧語を残さない**。「近接」「以内」「適切に」等は境界を数値・例で固定。
3. standard で既定通りのタスクは薄い指示でよい(モデルの記憶が効く)。過剰スペックは無駄。

### 振り分け表
| タスク | 指示の書き方 |
|---|---|
| standard・既定通り | 素の1行〜軽い仕様+出力契約(例: LRU、semver) |
| 曖昧 | そのまま投げない。発注側が出力項目まで固めてから投げる |
| 非既定ルールを含む | **急所ブロックで逸脱点を全て名指し+一例**。これさえ書けば subtle/反直感でも解ける |

## ステップ1.5: 既定プロファイルで「明示すべき点」を機械的に特定する(任意だが強力)
「どこまで細かく書くか」は勘で決めない。対象モデルの**既定プロファイル**を測れば、
明示すべき判断点が機械的に出る。`scripts/default_probe.py` がそれをやる。

- 仕組み: 「1点だけ未指定の極小タスク」を N 回投げ、生成コードを実行してどの既定を選んだか分類。
  → 各判断点の「既定」と「安定性(=何回中同じ選択をしたか)」が出る。適応リサンプリングで
  曖昧帯(0.5〜0.85)だけ自動的に追加サンプルして精密化。
- 使い方: `python3 scripts/default_probe.py <model> <base_url> <N> [think] [--api openai|anthropic] [--key KEY] [--only id1,id2]`
  model は省略可(OpenAI互換のみ): 先頭引数がURLなら `/v1/models` から自動検出
  (例: `default_probe.py http://localhost:8000 5`)。`""` プレースホルダでも同じ。
- **任意エンドポイント対応**: 既定はOpenAI互換(/v1/chat/completions — vLLM/llama.cpp/ollama等)。
  `--api anthropic` で Anthropic Messages形式(/v1/messages)。認証は `--key` か環境変数
  (PROBE_API_KEY/ANTHROPIC_API_KEY/OPENAI_API_KEY)。モデル差異(enable_thinking非対応、
  temperature拒否等)はクライアントが自動吸収。
- モデル比較: プロファイルは JSON 保存される。
  `python3 scripts/default_probe.py --diff A.json B.json` で「モデルを替えたら書き換える点」が出る。

### 明示要否の2条件(プロファイルから導く)
判断点を明示すべきなのは次のどちらか。両方満たさなければ書かなくてよい(過剰スペックは無駄):
1. **安定性が低い**(例 <0.8): モデルが既定を持たず実行ごとに揺れる → 必ず明示。
2. **既定が意図と食い違う**: 安定していても、既定の中身があなたの意図と違えば明示。
   ★ここが罠。「揺れないから省略」は危険。例(Qwen3.6-27B 実測):
   - `nth_index`=0-indexed で安定 → 「n番目」を1始まりで意図するとズレる
   - `sort_case`=ASCII順(大文字先)で安定 → 大小無視を期待するとズレる
   - `range_incl`=終端含む で安定 → range(終端除外)の直感と逆
   - `missing_key`=KeyError で安定 → None返しを期待するとズレる
   プロファイルは①を自動検出し、②は「既定の中身」を提示するので発注側が照合できる。
3. さらに「実装不能」カテゴリ(多数派がエラー)は明示しても救えない → 委譲回避か粒度昇降。

## ステップ1.6: タスク駆動スペック穴検出(bespoke・重要タスクに推奨)
固定バッテリー(1.5)はタスク固有の曖昧さを拾えない。`scripts/spec_holes.py` は
**ドラフト仕様そのもの**をワーカーに K 回実装させ、同じ入力で全実装を実行して
挙動が割れた入力=仕様の穴を機械検出する(disagreement probing)。
曖昧さを発注側が想像する必要がなく、「割れた」という事実が穴の証拠になる。

- 使い方: `python3 scripts/spec_holes.py <draft.txt> <関数名> <model> <base_url> 5 <inputs.json> [--api ...] [--key ...]`
  model は省略可(OpenAI互換のみ): 第3引数がURLなら `/v1/models` から自動検出。
  inputs.json(プローブ入力)は**発注側が用意する**(境界を突く引数の組の配列)。
  ワーカー提案の合併もあるが、弱いモデルは壊れたJSONを返すので発注側供給が確実。
- 出力の読み方(3シグナル):
  1. **[発散]** 実装間で挙動が割れた入力 → 仕様の穴。どちらが意図か決めて急所ブロックに固定
  2. **[合意]** 全実装が一致した暗黙挙動 → 一覧を見て意図と照合するだけ(違えば明示)
  3. **実装不能率** → 高い(50%超)ならこのタスク×モデルは委譲回避か粒度昇降
- 実証: 「上位n件」のドラフト仕様で、強い操作者だけが推論で捕まえられていた
  ソート/先頭の曖昧さを機械検出。= 新規曖昧さの検知が操作者の能力に依存しなくなる。

## ステップ2: 5ブロックで指示を書く(単発委譲)
`templates/task_prompt.md` を埋める。順序と各ブロックの意図:

1. **目的**(1文): 何を作るか。**ファイル名まで指定**(探索迷子を防ぐ)。
2. **仕様**(番号付き): 入力形式・出力項目・制約を全列挙。**書かないことは弱LLMが自己判断で埋める**。
3. **決定権の範囲**: 「自由に決めてよい / 厳守」を分ける。関数シグネチャ・戻り値・エラー型は厳守側へ。
4. **急所(bespokeなら必須)**: 罠になる境界条件を名指し + **入出力例を複数**。
   推測で外しうる箇所を具体例で固定する。standard タスクなら省略可。
5. **自己検証**: 「test_X.py に pytest を N ケース書き、全通過を確認」。弱LLMに自分でテストを
   書かせ実行させる(実測: 発注側の検証コストが下がる)。**隠しテストの内容はリークさせない**。
6. **出力契約**: 「通ったら DONE とだけ」。余計な自己解説を抑え出力トークンを節約。

## ステップ3: 独立検証(発注側の責務)
- 弱LLMの「DONE」「N passed」を信じない。**別に用意した隠しテスト**を当てる。
- bespoke タスクでは、弱LLMが書いた自作テストは自分の(誤った)解釈を検証しているだけの
  ことがある。境界条件は必ず発注側の隠しテストで突く。
- 失敗したら: 急所ブロックに外した境界を1つ足して再投下(粒度を最小限だけ上げる)。

## アンチパターン(実測で確認)
- 探索迷子: 曖昧だと ls から延々さまよう → 目的とファイル名を先頭で固定。
- 暗黙の自己判断: 決定権が不明だと検証不能な成果物になる → ブロック3で範囲を切る。
- 過剰実装: 指定がないと余計な機能・行数を盛る → 仕様を「これだけ」と閉じる。
- 罠の推測外し(bespoke): 境界を書かないと弱LLMは片側を勝手に決める → ブロック4で例示。

## 発注側(上位モデル)がやること・やらないこと
- やる: 分類 → スペック化 → 独立検証 → 失敗診断。スペック作成は数千tokで安く、実装の
  数十万tok(弱LLM単価)を置き換えるので割に合う。
- やらない: プロダクトコードを自分で書く(それは委譲の意味を失う)。

## 補足: このスキル自体の操作者モデル依存性(検証済み)
判断を「機械的手順+データ(プロファイル)」に外部化しているため、核心の照合
(既定vs意図の突き合わせ)は Haiku 級の操作者でも遂行できることを実測確認済み。
プロファイルに無い新規曖昧さの検知は spec_holes.py が機械化する。
弱い操作者ほどプローブ点を厚くして、操作者自身の推論に委ねる部分を減らすとよい。

実測の詳細(実験ログ・数値)はリポジトリの docs/FINDINGS.md を参照。
